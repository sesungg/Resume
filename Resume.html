Troubleshooting

Spring 트랜잭션에서 데이터 조회 문제

REQUIRES_NEW 트랜잭션에서 삽입한 데이터가 조회되지 않는 문제 해결

문제 상황
Spring 기반 애플리케이션에서 @Transactional(readOnly = false)가 적용된 A 서비스의 a 메서드 내부에서,
@Transactional(propagation = Propagation.REQUIRES_NEW)가 적용된 B 서비스의 b 메서드를 호출해 테이블에 데이터를 삽입했습니다.
하지만 b 메서드 실행 직후, a 메서드에서 해당 테이블을 조회했을 때 삽입된 데이터가 반영되지 않는 문제가 발생했습니다.

트랜잭션 흐름 시각화

                        graph LR
                            A["A Service\n@Transactional(readOnly=false)"] -->|1. 호출| B["B Service\n@Transactional(propagation=REQUIRES_NEW)"]
                            B -->|2. INSERT| C[테이블]
                            A -->|3. SELECT| C
                            C -->|조회 실패| D["격리 수준 문제"]
                            D --> E["Isolation.READ_COMMITTED 설정"]
                            E --> F["성공적으로 데이터 조회됨"]

문제 원인 분석

  • 트랜잭션 전파: b 메서드는 REQUIRES_NEW 설정으로 인해 기존 트랜잭션(a 메서드)과는 별도의 새로운 트랜잭션에서 실행되며, 독립적으로 커밋됩니다.
  • 데이터베이스 격리 수준: 데이터베이스의 격리 수준이 REPEATABLE READ로 설정되어 있어, a 메서드 트랜잭션이 시작된 시점의 스냅샷을 기준으로 데이터를 조회하게 됩니다.
    이로 인해 b 메서드에서 커밋된 새로운 데이터는 a 메서드의 트랜잭션에서는 조회되지 않습니다.

해결 방법

  • a 메서드의 트랜잭션에 @Transactional(isolation = Isolation.READ_COMMITTED)를 명시적으로 설정하여, 다른 트랜잭션에서 커밋된 데이터를 즉시 조회할 수 있도록 조정했습니다.

결과
트랜잭션 격리 수준을 READ_COMMITTED로 설정한 이후, a 메서드에서도 b 메서드에서 삽입한 데이터를 정상적으로 조회할 수 있게 되었습니다.
문제의 원인을 트랜잭션 전파와 격리 수준에 집중해 정확히 분석했으며, 불필요한 캐시 조작이나 코드 변경 없이 효율적으로 해결할 수 있었습니다.

배운 점

  • Spring 트랜잭션의 전파(Propagation)와 데이터베이스 격리 수준(Isolation)이 실질적으로 애플리케이션의 데이터 정합성과 직결된다는 점을 경험적으로 이해했습니다.
  • 문제가 발생했을 때 단순한 코드 수정보다는 트랜잭션 흐름과 DB 설정을 먼저 분석하는 것이 중요하다는 교훈을 얻었습니다.
  • 추후 유사한 트랜잭션 문제를 예방하기 위해 기본 격리 수준을 명확히 인지하고, 서비스 설계 시 트랜잭션 전파와 조회 타이밍을 신중하게 고려할 수 있게 되었습니다.

QueryDSL 동적 조건 처리 중 IllegalArgumentException 방지

검색 조건이 null일 경우 쿼리 빌더에서 예외 발생 → null-safe 처리로 해결

문제 상황
관리자 페이지 마일리지 내역 검색에서 여러 조건을 입력받아 QueryDSL로 조건 검색을 구현하던 중,
일부 조건이 null일 경우 BooleanExpression 내부에서 IllegalArgumentException이 발생하는 문제가 있었습니다.

해결 방법

  • 관련 문제를 인프런 질문 [참고 링크 🔗]를 통해 확인하고,
    BooleanBuilder를 안전하게 생성할 수 있도록 nullSafeBuilder() 헬퍼 메서드(BooleanBuilder + 람다 기반 null-safe 조건 메서드)를 도입해 해결 

private BooleanBuilder ageEq(Integer age) {
    return nullSafeBuilder(() -> member.age.eq(age));
}

private BooleanBuilder roleEq(String roleName) {
    return nullSafeBuilder(() -> member.roleName.eq(roleName));
}

public static BooleanBuilder nullSafeBuilder(Supplier<BooleanExpression> f) {
    try {
        return new BooleanBuilder(f.get());
    } catch (IllegalArgumentException e) {
        return new BooleanBuilder(); // 안전하게 무시
    }
}

결과
null 입력이 있어도 쿼리 빌드 과정에서 예외가 발생하지 않으며, 모든 조건이 안전하게 동작하도록 개선되었습니다.
코드의 반복도 줄이고 동적 조건을 더 유연하게 처리할 수 있게 되었습니다.

배운 점

  • QueryDSL에서 null-safe 조건 처리를 위한 헬퍼 메서드가 코드 안정성 및 유지보수성에 매우 효과적임을 경험
  • BooleanBuilder, Supplier, 람다를 활용한 함수형 null 처리 패턴의 장점을 실감

Architecture & Pattern

다형성과 전략 패턴을 이용한 메시지 발송 시스템 설계

NHN Cloud 메시지 API (SMS, LMS, Email, 알림톡) 대응

  • 배경: NHN Cloud의 메시지 API는 유형별로 명세가 달라 중복된 조건 분기 코드가 많았음
  • 해결 전략: 전략 패턴(Strategy Pattern)을 적용해 메시지 발송 로직을 추상화
  • 구성 클래스:
    • SendStrategy: 메시지 전송 전략의 인터페이스
    • SMSSendStrategy: SMS 전송 구현
    • LMSSendStrategy: LMS 전송 구현
    • EmailSendStrategy: 이메일 전송 구현
    • SendContext: 전략을 주입받아 실행하는 컨텍스트 클래스
  • 확장성 고려: 향후 알림톡 추가를 위해 및 새로운 업체로 변경을 위해 SendStrategy를 상속하는 새 전략 클래스만 구현하면 됨 
                        classDiagram
                            class SendStrategy {
                                <<interface>>
                                +send(Message): void
                            }

                            class SendContext {
                                -strategy: SendStrategy
                                +setStrategy(strategy: SendStrategy): void
                                +execute(message: Message): void
                            }

                            class SMSSendStrategy {
                                +send(message: Message): void
                            }

                            class LMSSendStrategy {
                                +send(message: Message): void
                            }

                            class EmailSendStrategy {
                                +send(message: Message): void
                            }

                            class AlimtalkSendStrategy {
                                +send(message: Message): void
                            }

                            SendStrategy <|.. SMSSendStrategy
                            SendStrategy <|.. LMSSendStrategy
                            SendStrategy <|.. EmailSendStrategy
                            SendStrategy <|.. AlimtalkSendStrategy
                            SendContext --> SendStrategy

  • 전략 패턴을 통한 유연한 메시지 발송 시스템 구현 경험 축적
